# Daily Operating System Patterns

## Definition

The recurring structural patterns that govern sustainable, high-leverage daily knowledge work. These are not productivity tips — they are operational principles derived from cognitive science and system design, applied to a single-practitioner workspace.

## Why It Matters

Daily execution quality determines whether a knowledge system compounds or stagnates. The workspace can have perfect architecture, 80+ skills, and excellent knowledge base — but if daily execution is reactive, depleted, and context-overloaded, the system produces noise instead of intelligence. The Daily OS patterns are the operational substrate.

## Key Patterns

**Pattern 1: Minimum Viable Ritual**
The ritual set should be the minimum that produces observable change in execution quality. Adding rituals is easy; each addition has a carrying cost. The test: if you skipped this ritual for a week, would your execution quality measurably degrade? If no: remove it.

Current minimum viable set: `/briefing` (orient) + `/shutdown` (close). Everything else is additive.

**Pattern 2: Energy-Aware Routing**
Cognitive work has three energy states: high (produce), medium (process), low (file and clear). Misrouting energy — doing high-demand work in a low-energy state — produces low-quality output with high time cost. The output of a depleted deep work session is often worse than no output at all.

Energy routing is not a fixed schedule. It's a real-time assessment: "What is my cognitive state right now? What work matches that state?"

**Pattern 3: Top 3 Rule**
Three commitments per day, maximum. Not a to-do list. Three. The difference: a to-do list distributes attention; three commitments concentrate it. Three done > eight partially done. The prioritization hierarchy (commitment → leverage → reversibility) determines which three.

**Pattern 4: Context Ceiling**
Maximum 3 files loaded before starting any task. Beyond 3, context loading becomes context fragmentation — attention spreads across sources rather than concentrating on the problem. If a task seems to require >3 files: the task scope is wrong, or the context architecture needs work.

**Pattern 5: EOD Shutdown as Cognitive Closure**
Unfinished tasks remain active in working memory (the Zeigarnik effect). EOD shutdown creates deliberate cognitive closure: captures opens, seeds tomorrow's first action, closes the loop. Without it, the day's open threads stay active overnight and degrade next-session startup quality.

The "tomorrow starts with" seed is the single most leveraged output of the shutdown ritual. Pre-loading the first action eliminates the startup cost of "what do I do first?"

**Pattern 6: One-Sentence Context Switch Record**
Before switching contexts: write one sentence. Current state + next action. This costs 30 seconds and saves 5-15 minutes of context reconstruction per switch. Across 4 switches per day, that's 20-60 minutes of recovered time.

**Pattern 7: Done Condition Before Work**
Every deep work session should have a named done condition before starting. Without it: scope drift, energy dissipation, and an undefined endpoint. With it: clear stopping criteria, completable in a bounded time block. Done conditions force scope scoping before momentum builds.

**Pattern 8: Session Scope Bounded at 60-90 Minutes**
Deep work sessions longer than 90 minutes produce diminishing returns for most knowledge work types. The exception: creative flow states in writing or design. The rule for planning: design sessions for 60-90 minutes. Chain sessions with breaks rather than running one long block.

**Pattern 9: Meeting Preparation Threshold**
Not every meeting warrants preparation. The threshold: does preparation change the outcome? High-stakes decisions, strategy reviews, escalations, customer conversations: yes. Routine standups: no. Applying prep universally reduces its impact by diluting the attention it receives.

**Pattern 10: Compounding via Daily Capture**
Daily `/capture` inputs compound into weekly `/weekly` synthesis, which compounds into monthly cognitive review, which compounds into the knowledge base. The daily capture is the entry point. Without daily capture discipline, the compounding chain breaks at the first link.

## In Practice

**Diagnostic questions for daily operations:**
- Did the top 3 commitments match what actually got done? (If consistently no: planning quality issue)
- How many context switches happened? (>6/day: context fragmentation)
- Was EOD shutdown run? (If consistently skipped: open loop accumulation)
- What was the energy state when the hardest work happened? (If consistently low: energy routing issue)

**Weekly review signal:**
At the end of each week, check the week's actual theme against the planned theme. Consistent divergence between planned and actual theme means: either planning is disconnected from reality, or reactive work is systematically overwhelming planned work. Either is a signal to address — not just accept.

## Common Failure Modes

| Failure | Description | Pattern Violated |
|---|---|---|
| Ritual inflation | Adding more rituals for more structure | P1 — Minimum Viable Ritual |
| Energy mismatch | Hard thinking when depleted | P2 — Energy-Aware Routing |
| List sprawl | 12-item to-do lists instead of top 3 | P3 — Top 3 Rule |
| Context overload | 6+ files per task | P4 — Context Ceiling |
| Open loop accumulation | Skipping shutdown, threads stay active | P5 — EOD Shutdown |
| Context switch blindness | Switching without recording state | P6 — One-Sentence Record |
| Vague sessions | Starting deep work without done condition | P7 — Done Condition First |
| Marathon sessions | 3-hour deep work blocks | P8 — 60-90 Minute Scope |
| Universal meeting prep | Preparing for every meeting | P9 — Preparation Threshold |
| Capture gaps | Insights lost before weekly review | P10 — Daily Capture |
